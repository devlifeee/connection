# Связь — Система децентрализованной связи

## О проекте

**Hex.Team** — это автономная система для децентрализованного обмена сообщениями, файлами и совершения аудио/видеозвонков. Проект создан для обеспечения связи в условиях отсутствия интернета или централизованной инфраструктуры (аварийные ситуации, перегрузка сетей, изолированные зоны).

### Контекст и цель
В современном мире связь критически зависит от вышек сотовой связи и облачных серверов. **Hex.Team** решает проблему коммуникации, создавая Mesh-сеть, где каждое устройство выступает в роли клиента и ретранслятора.
- **Работа без интернета**: полная независимость от провайдеров.
- **P2P соединение**: прямая передача данных между устройствами.
- **Безопасность**: сквозное шифрование и отсутствие центрального сервера, который мог бы собирать данные.

### Ключевые возможности
- **Обнаружение узлов**: автоматический поиск устройств в локальной сети (mDNS) и через DHT.
- **Текстовые сообщения**: мгновенный обмен зашифрованными сообщениями с подтверждением доставки.
- **Аудио и Видеозвонки**: P2P звонки с низким джиттером и задержкой (WebRTC).
- **Передача файлов**: отправка файлов любого размера с контролем целостности (хэш-суммы) и отображением прогресса.
- **Мультихоп-архитектура**: заложена логика ретрансляции сообщений через промежуточные узлы.
- **Кроссплатформенность**: Веб-интерфейс (React) + Go-агент, работающий на Windows, Linux, macOS.

## Архитектура и Технологии

### Общая схема
Проект построен по клиент-серверной архитектуре, где "сервером" выступает локальный **Node Agent** (Sidecar), запускаемый на устройстве пользователя. Frontend общается только с локальным агентом, который берет на себя всю сложность P2P-взаимодействия.

### Frontend (Client)
Отвечает за визуализацию и взаимодействие с пользователем.
*   **Framework**: React 18 + Vite + TypeScript.
*   **State Management**: TanStack Query (кэширование серверных данных) + React Context (локальное состояние сессии).
*   **UI Kit**: TailwindCSS + Radix UI (доступные примитивы) + Lucide Icons.
*   **Media**: Native WebRTC API (управление камерой/микрофоном, рендеринг видеопотоков).
*   **Взаимодействие с агентом**:
    *   **REST API**: Для отправки команд (начать звонок, отправить сообщение, настройки).
    *   **WebSocket**: Для получения событий в реальном времени (входящий звонок, новое сообщение, обновление списка пиров).

### Backend (Node Agent)
Автономный сервис на Go, реализующий логику P2P-узла.
*   **Language**: Go 1.24.
*   **Network Stack**: **libp2p** — де-факто стандарт для децентрализованных сетей (используется в IPFS, Ethereum 2.0).
    *   **Transport**: TCP, WebSocket, QUIC (UDP).
    *   **Security**: Noise handshake (шифрование соединений).
    *   **PubSub**: GossipSub для обмена сообщениями о присутствии.
    *   **Relay**: Circuit Relay v2 для обхода NAT.
*   **Storage**: **PostgreSQL** (через драйвер `pgx/v5`) — надежное хранение истории сообщений, контактов и метаданных файлов.
*   **API**: Собственный HTTP-сервер на базе `net/http` и `gorilla/websocket`.
*   **Модули**:
    *   `discovery`: mDNS и DHT.
    *   `media`: Сигнализация звонков (обмен SDP/ICE через P2P-стримы).
    *   `chat`: Обработка сообщений, очередей и ACK.
    *   `filetransfer`: Потоковая передача файлов.

### Стек технологий
- **Frontend**: React, Vite, TypeScript, TailwindCSS, Lucide Icons.
- **Backend (Node Agent)**: Go (Golang), libp2p.
- **Транспорт**: TCP/WebSockets, QUIC (экспериментально).
- **Сигнализация и NAT**: mDNS (локально), DHT (распределенная хеш-таблица), STUN/TURN (для звонков).
- **Звонки**: WebRTC (API браузера + Go signaling).

### Топология сети
Система строит **неструктурированную P2P-сеть** с элементами Mesh.
1.  **Discovery (Обнаружение)**: 
    *   **Local Discovery (mDNS)**: Узлы автоматически находят друг друга в одной Wi-Fi сети без настройки серверов.
    *   **Global Discovery (DHT)**: Используется Kademlia DHT (Distributed Hash Table) для поиска узлов за пределами локального сегмента, если они имеют публичные IP или проброшенные порты.
2.  **Маршрутизация и Relay**:
    *   Если прямое соединение невозможно (например, оба узла за строгим NAT), система использует протокол **Circuit Relay v2**. Трафик инкапсулируется и передается через доступный промежуточный узел (Relay), который "видит" обоих участников.
3.  **Устойчивость**:
    *   Сеть самоорганизуется: при отключении одного узла остальные продолжают взаимодействовать.
    *   Поддерживается **Connection Manager**, который контролирует количество активных соединений, закрывая неиспользуемые для экономии ресурсов.

## Детальное описание реализации

### 1. Авторизация и Идентификация
В децентрализованной сети нет логинов и паролей.
*   **Identity**: При первом запуске агент генерирует криптографическую пару ключей (Ed25519).
*   **PeerID**: Уникальный идентификатор узла, являющийся хэшем от его публичного ключа (например, `12D3KooW...`).
*   **Аутентификация**: При каждом соединении происходит процедура handshake (протокол **Noise**), где узлы обмениваются публичными ключами и доказывают владение приватным ключом. Подмена PeerID невозможна, так как злоумышленник не сможет подписать challenge без приватного ключа.
*   **Display Name**: Для удобства пользователей поверх PeerID работает протокол **Identify/Presence**, позволяющий привязать к техническому ID читаемое имя (никнейм), которое распространяется через GossipSub.

### 2. Протокол сообщений (Reliability)
Для обеспечения надежной доставки сообщений в нестабильной среде реализован кастомный протокол поверх libp2p stream:
*   **Структура пакета**: `[Length (4 bytes)][Protobuf Message]`.
*   **ID сообщения**: UUID v4 генерируется на клиенте.
*   **Дедупликация**: Каждый узел хранит LRU-кэш обработанных MessageID. Повторно пришедшие пакеты (например, через разные пути в mesh-сети) отбрасываются.
*   **Подтверждения (ACK)**: Получатель обязан отправить техническое сообщение `ACK` с ID принятого пакета. Если отправитель не получает ACK в течение таймаута, запускается механизм **Exponential Backoff Retry**.
*   **Защита от петель (TTL)**: В заголовке пакета есть поле `HopLimit`, уменьшающееся при каждой ретрансляции.

### 3. Звонки в реальном времени (Real-time)
Реализована гибридная схема для минимизации задержек:
*   **Сигнализация (Signaling)**: Обмен SDP (Session Description Protocol) и ICE кандидатами происходит через защищенный канал libp2p (тот же, что и для сообщений). Это позволяет установить звонок даже через Relay.
*   **Медиа-траспорт (WebRTC)**:
    *   Браузер захватывает медиа (`navigator.mediaDevices.getUserMedia`).
    *   Используется **UDP** (через STUN) для передачи RTP пакетов, что критично для real-time (TCP дает задержки при потере пакетов).
    *   **Jitter Buffer**: Встроенный в WebRTC механизм компенсирует неравномерность прихода пакетов.
    *   **Trickle ICE**: Кандидаты отправляются по мере обнаружения, не дожидаясь сбора всех, что ускоряет начало звонка на 1-2 секунды.
*   **Fallback**: Если P2P WebRTC соединение не проходит (строгий NAT), звонок не состоится (в текущей версии TURN не развернут, но архитектурно поддерживается).

### 4. Передача файлов
- Файлы разбиваются на чанки.
- Для каждого файла генерируется ID и хэш для проверки целостности.
- UI отображает прогресс-бар загрузки.

### 4. Безопасность
- **Шифрование трафика**: Все P2P соединения шифруются (Noise Secure Channel).
- **Аутентификация**: Узлы идентифицируются по публичным ключам (PeerID).
- **Приватность**: IP-адреса скрыты за абстракцией PeerID (кроме этапа прямого соединения).

## Документация по запуску

### Предварительные требования
- **Go** 1.21+
- **Node.js** 18+
- **npm** или **yarn**

### Запуск Node Agent (Backend)
Агент отвечает за сетевое взаимодействие.
```bash
cd node-agent
go run cmd/main.go
# Агент запустится на порту 8080 (HTTP API) и 4001 (P2P)
```

### Запуск Frontend (UI)
Интерфейс пользователя.
```bash
npm install
npm run dev
# Приложение доступно по адресу http://localhost:5173
```

### Демонстрация (Multi-node)
Для запуска нескольких узлов на одной машине используйте скрипт:
```bash
npm run multi
# Запустит 3 независимых агента и 3 клиента React
```

## Тестирование и Надежность

В проекте реализованы механизмы повышения надежности:
- **Retry Policy**: Автоматическая переотправка сообщений при сбое.
- **Heartbeats**: Периодическая проверка доступности пиров.
- **Error Boundaries**: React-компоненты защищены от падения всего интерфейса при ошибках рендеринга.
- **Logs**: Подробное логирование сетевых событий в консоль агента и браузера.
